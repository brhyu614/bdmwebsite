# 11 · UI/UX 원칙 (사용자 피드백에서 학습됨)

> 이 문서는 BDM Lab Studio 베타 개발 중 교수님이 반복적으로 지적한 UI/UX 원칙을 정리한 것이다.
> **새 세션에서 Studio·프론트 작업 전에 반드시 읽는다.**

## 1. 동등한 선택지는 시각적 무게도 동등하게

사용자에게 "고르라"고 제시한 옵션은 크기·스타일·비주얼 무게를 같게 한다.

- ❌ 기본 경로는 큰 카드, 대안은 작은 ghost 버튼/링크
- ✅ 스타일 카드, "+ 새 스타일 만들기", "스타일 없이 만들기" **셋 다 동일 그리드·동일 크기 카드**

**이유:** 기본 경로를 무의식적으로 강조하면 사용자는 "왜 이 선택지를 만들어놨지?" 느낌을 받음. Studio Step 1에서 두 번 지적됨.

## 2. 사용자의 정당한 선택에 대해 부정/경고 문구 금지

- ❌ "(품질은 낮아집니다)", "⚠ 품질 낮음", "제한적입니다", "추천하지 않습니다"
- ✅ "스타일 없이 진행", "✓ 선택됨" — 중립·긍정 톤

경고 언어는 **예외 상황**(진짜 에러, 파괴적 액션 직전)에만. 정상 선택지에 딱지 붙이지 않는다.

## 3. 첫 화면 = 눈에 띄는 핵심 액션부터

로그인 직후 바로 핵심 선택을 정면에 보여준다. 드롭다운 안에 숨기거나 "위에 눈에 안 띄게" 두면 안 됨.

- Studio: 로그인 직후 "어떤 스타일로 만들까요?" + 스타일 카드 그리드가 **스텝 네비보다 먼저** 시선을 끌어야 함
- styles.html 같은 관리 페이지는 네비 링크로만 접근, 메인 흐름 중간에 끼우지 않음

## 4. 기술 용어를 UI에 그대로 노출 금지

- ❌ "429 / rate limit / retry_after / throttle"
- ✅ "이미지 서버가 잠깐 느려지면 자동으로 기다렸다가 이어서 생성"

개발자 디버깅용 로그에는 원래 용어를 써도 되지만, 사용자 눈에 보이는 다이얼로그·알림·버튼 라벨은 한국어 친절어로.

## 5. 자유입력보다 선택지 제시가 우선

카테고리·톤·분량 같은 항목은 프리셋 chip 버튼으로 먼저 제공하고, "기타 (직접 입력)"을 추가 옵션으로 둔다.

**이유:** 빈 input에 "예: 학술적 / 가벼운"만 놔두면 사용자는 뭘 써야 할지 모르고 기본값으로 넘어감. 선택지가 있으면 브랜드 일관성도 쌓이고 집단지성 데이터도 누적됨.

## 6. 투명성: "진짜 스타일이 반영되고 있어?"

사용자는 보이지 않으면 믿지 않는다. AI에 무엇을 보냈는지 **직접 확인 가능**해야 함.

- Step 3 상단에 지금 주입 중인 스타일 요약 배너
- 각 AI 생성물에 "🔍 프롬프트 보기" 버튼 — Replicate에 실제 보낸 full prompt 표시
- 교수님이 명시적으로 "난 너가 정말 스타일대로 만들고 있는지 모르겠어"라고 했음. 코드상 주입되는 것만으로 부족함

## 7. 이미지 첨부 이슈

교수님이 스크린샷 첨부하다 "Image was too large" 에러로 쓰레드가 자주 끊김. 스트레스 누적됨.

**대응:**
- 스크린샷 필요할 때는 "가능하면 작게 압축해서 보내달라" 한 번만 안내
- 텍스트 묘사로 우회 가능하면 그쪽 유도
- 쓰레드 끊겨도 `_memory/08-current-state.md` 읽으면 즉시 복귀 가능하게 유지

## 8. 이미 된 작업 재요구 금지

`feedback_state_persistence` 원칙과 연결 — API 키·마이그레이션·함수 배포 같은 "이미 설정된 것"을 다시 설정하라고 요구하지 말 것. `npx supabase@latest secrets list` / `functions list`로 내가 먼저 확인해서 답을 찾는다.

## 9. 설명·브리프는 짧게, 선택지 나열 금지

`01-user.md` 커뮤니케이션 원칙과 동일. "A안/B안 중 고르세요" 대신 기본 방향 정하고 실행. 엇나간 부분은 사용자가 짚으면 그때 수정.
