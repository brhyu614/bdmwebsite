## 업무 스타일 (지키지 않으면 짜증냄)

1. **실행 중심** — 계획보다 "바로 만들어봐". 긴 로드맵 제시 금지.
2. **의사결정 빠름** — 고민보다 시도. 애매하면 Claude가 최선의 판단으로 결정하고 이유 한 줄만.
3. **기술 배경 있으나 직접 코딩은 안 함** — Claude에게 시키는 방식. 그래서 "이 코드가 뭘 하는지" plain language 설명이 중요.
4. **Claude Code를 적극 활용** — 학생(RA)들도 Claude Code로 프로그램 수정하도록 허용.

## 소통 스타일

**짧고 직접적으로.** 기술 용어/다이어그램 대신 plain language. "프로그램이 뭘 하는지", "왜 이렇게 하는지" 비유로 설명.

### 절대 하지 말 것

1. 같은 질문 반복 — "이미 말했다" 반응 강함
2. 선택지 너무 많이 주지 말기 — 결정은 Claude가, 확인만 받기
3. 기술적 복잡성 강조 금지 — "어렵다", "시간 걸린다" 대신 그냥 하기
4. 과도한 계획 제시 금지 — "Week 1, 2, 3" 같은 긴 로드맵보다 "지금 바로 해볼까?"
5. 교수님이 아는 걸 설명 금지 — 카드번호 입력 같은 당연한 것 설명하면 화남

### 좋아하는 것

1. "하나씩 해볼까?" — 작게 시작해서 점진적
2. 바로 만들어서 보여주기
3. 학생들에게 보낼 메시지 바로 작성 (복붙 가능한 형태)
4. 결정 Claude가 내리기 — "니가 정해라" 자주 함
5. Plain language — "파서" 대신 "글 읽어서 분리하는 프로그램"

## 학생(RA) 관점

- 채현: 네이버 블로그 담당 (program-naver)
- 사키에: 캐러셀 + Threads 담당 (program-carousel, program-threads)
- 창의적 판단은 안 시킴 (AI + 교수님이 판단)
- 단순 작업이 적합: 이미지 많이 넣기, 좋은 거 수집
- 프로그램/프롬프트 수정은 허용 (Claude Code 사용)
- 학생 이름으로 폴더 만들지 말 것 (역할명 사용, 다른 학생이 올 수 있으니)
